# MILESTONES & 버전 관리

> 이 문서는 **배포 전 버전 관리 규율**과 **마일스톤 로드맵**을 담습니다.
> 상세 변경 내역 → [`CHANGELOG.md`](./CHANGELOG.md) · 코드 밖 기획/분석 → `../AI앱개발/`
> **모든 프로덕션 배포(`git push origin main`) 직전에 이 문서와 CHANGELOG·버전을 함께 갱신합니다.**

---

## 🔢 버전 규칙 (SemVer — `vMAJOR.MINOR.PATCH`)

| 자리 | 올리는 경우 | 예시 |
|------|-------------|------|
| **MAJOR** (`X.0.0`) | 플랫폼 정체성·구조의 큰 전환, 비호환 재편 | v3 멀티언어 플랫폼, v4 AI 앱 개발 코너 |
| **MINOR** (`x.Y.0`) | 새 트랙/강의 세트/주요 기능 추가 (하위호환) | 중급1 트랙 8강, 새 워크샵 Phase, 새 페이지·모드 |
| **PATCH** (`x.y.Z`) | 버그 수정·콘텐츠 정정·문구/공지 갱신·소규모 개선 | 강의 사실 정정, 공지 갱신, 핫픽스 |

**원칙**
- **버전 없는 배포 금지.** 모든 프로덕션 배포는 최소 PATCH 범프 + CHANGELOG 항목 + git 태그를 남긴다.
- 한 릴리즈에 여러 성격이 섞이면 **가장 높은 자리** 기준으로 범프 (예: 기능 추가 + 버그 수정 → MINOR).
- `package.json`의 `version`, git 태그 `vX.Y.Z`, CHANGELOG의 `[X.Y.Z]` 는 **항상 일치**해야 한다.

---

## ✅ 배포 전 체크리스트 (매 배포 직전 실행)

```
1. npm run build                 # tsc + vite 통과 필수 (codegen 포함)
2. 범프 결정                      # 변경 성격 → MAJOR / MINOR / PATCH
3. package.json "version" 갱신
4. CHANGELOG.md                  # [Unreleased] → ## [X.Y.Z] - YYYY-MM-DD
                                 #   분류: Added / Changed / Fixed / Security / Removed
5. MILESTONES.md                 # 아래 '릴리즈 로그'에 한 줄 + 로드맵 체크박스 갱신
6. git commit -m "release: vX.Y.Z — 한 줄 요약"
7. git tag vX.Y.Z
8. git push origin main && git push origin vX.Y.Z
9. 헬스체크                       # 메인 200 / 함수 부팅(api/* 500 없음) / 번들 해시 갱신
```

> 작은 콘텐츠 정정·핫픽스도 **PATCH 범프 + 태그**를 남긴다. "별것 아니라" 건너뛰지 않는다.
